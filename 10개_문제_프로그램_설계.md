작성 기준: 비개발자, 초등학생도 이해할 수 있는 수준

## 문제 1. AI 기반 Git 커밋 & PR 자동 생성기

### 1. 무엇을 만드는가

내가 코드를 고친 내용을 자동으로 읽고, AI가 “무엇을 고쳤는지”를 정리해 주는 프로그램을 만든다.  
그리고 Git 커밋 메시지와 GitHub PR 설명문도 자동으로 써 준다.

### 2. 왜 필요한가

개발자는 코드를 고친 뒤 “이번에 무엇을 바꿨는지”를 글로 남겨야 한다.  
하지만 매번 정리하는 일이 귀찮고 빠뜨리기 쉽다.  
이 프로그램은 바뀐 내용을 읽고 정리문을 대신 만들어 준다.

### 3. 쉬운 비유

숙제를 고친 뒤 선생님에게 “어디를 왜 고쳤는지” 설명해야 한다고 생각하면 된다.  
AI는 내 공책을 보고 “이 학생은 제목을 고치고, 틀린 계산을 바로잡았습니다”라고 요약해 주는 친구이다.

### 4. 필요한 부품

- Git 변경 내용 읽는 부분
- AI에게 질문을 보내는 부분
- 커밋 메시지를 만드는 부분
- PR 설명문을 만드는 부분
- 결과가 규칙에 맞는지 확인하는 부분

### 5. 동작 순서

1. 사용자가 프로그램을 실행한다.
2. 프로그램이 `git status`로 어떤 파일이 바뀌었는지 확인한다.
3. 프로그램이 `git diff`로 파일 안의 어떤 내용이 바뀌었는지 읽는다.
4. 바뀐 내용을 AI에게 보낸다.
5. AI가 커밋 메시지와 PR 설명문을 만든다.
6. 프로그램이 결과를 보기 좋게 출력한다.

### 6. 화면 또는 결과물

결과는 터미널에 이렇게 나온다.

```text
Commit Message
feat: add login validation

PR Title
Add login validation

PR Body
Why
- 잘못된 로그인 입력을 막기 위해

What
- 로그인 입력 검증 추가

How to Test
- 빈 값으로 로그인 시도
```

### 7. 확인 방법

- 바뀐 파일이 없을 때 안내 문구가 나오는지 확인한다.
- API Key가 없을 때 친절한 오류가 나오는지 확인한다.
- 커밋 제목이 너무 길지 않은지 확인한다.
- PR 본문에 Why, What, How to Test가 있는지 확인한다.

### 8. 조심할 점

코드 안에 비밀번호나 API Key가 있을 수 있다.  
그래서 AI에게 보내기 전에 중요한 정보는 가려야 한다.

## 문제 2. 클라우드 환경에서 웹 서비스 인프라 구축

### 1. 무엇을 만드는가

인터넷에서 접속할 수 있는 웹 서버를 클라우드에 만든다.  
AWS 안에 안전한 네트워크 공간을 만들고, 그 안에 서버를 두고, 필요한 문만 열어 둔다.

### 2. 왜 필요한가

웹 서비스를 만들면 다른 사람이 인터넷으로 접속할 수 있어야 한다.  
하지만 아무 문이나 열어 두면 위험하다.  
그래서 안전한 길만 만들고, 허락된 사람만 관리할 수 있게 해야 한다.

### 3. 쉬운 비유

클라우드는 큰 건물이고, VPC는 내가 빌린 방이다.  
서버는 방 안에 있는 컴퓨터이고, 보안 그룹은 출입문 경비원이다.  
웹 접속 문은 열어 두지만, 관리자 문은 내 집 주소에서만 들어오게 한다.

### 4. 필요한 부품

- VPC: 내 전용 네트워크 공간
- Subnet: 서버가 들어갈 작은 구역
- Internet Gateway: 인터넷으로 나가는 문
- Route Table: 길 안내 표지판
- EC2: 실제 웹 서버 컴퓨터
- Security Group: 들어오는 요청을 막거나 허용하는 문지기
- IAM: AWS 안에서 누가 무엇을 할 수 있는지 정하는 권한

### 5. 동작 순서

1. AWS 서울 리전에 VPC를 만든다.
2. Public Subnet을 만든다.
3. Internet Gateway를 붙인다.
4. Route Table에 인터넷으로 나가는 길을 등록한다.
5. EC2 서버를 만든다.
6. 서버에 Nginx 같은 웹 서버를 설치한다.
7. HTTP 80번 포트는 열고, SSH 22번 포트는 내 IP만 허용한다.
8. 브라우저에서 서버 주소로 접속한다.

### 6. 화면 또는 결과물

- 웹 브라우저에서 `http://서버IP` 접속 성공
- 아키텍처 그림 1장
- 문제 해결 보고서
- 리소스 삭제 체크리스트

### 7. 확인 방법

- 서버 안에서 `curl localhost`가 성공하는지 확인한다.
- 내 컴퓨터 브라우저에서 서버 IP로 접속되는지 확인한다.
- 보안 그룹에 HTTP만 전체 허용되어 있는지 확인한다.
- SSH는 내 IP만 허용되어 있는지 확인한다.

### 8. 조심할 점

모든 포트를 열면 위험하다.  
실습이 끝나면 EC2, EBS, EIP 같은 자원을 지워야 과금이 생기지 않는다.

## 문제 3. 인증과 연관관계로 완성하는 FastAPI 웹 서비스

### 1. 무엇을 만드는가

로그인한 사람만 사용할 수 있는 웹 서비스를 만든다.  
또 사용자, 글, 댓글처럼 서로 연결된 데이터를 저장하고 보여 준다.

### 2. 왜 필요한가

웹 서비스에는 “누가 들어왔는지”와 “그 사람이 무엇을 할 수 있는지”가 중요하다.  
로그인하지 않은 사람이 남의 정보를 보면 안 된다.

### 3. 쉬운 비유

학교 도서관을 생각하면 된다.  
학생증이 있어야 책을 빌릴 수 있다.  
또 “학생”, “책”, “대여 기록”은 서로 연결되어 있다.

### 4. 필요한 부품

- 로그인 기능
- 로그아웃 기능
- 로그인 여부 확인 기능
- 사용자 데이터
- 서로 연결된 데이터 모델 3개 이상
- 상태를 바꾸는 기능
- HTML 화면

### 5. 동작 순서

1. 사용자가 로그인 화면에 들어간다.
2. 아이디와 비밀번호를 입력한다.
3. 서버가 맞는 사용자인지 확인한다.
4. 맞으면 로그인 상태를 저장한다.
5. 사용자는 보호된 페이지에 들어갈 수 있다.
6. 사용자는 데이터를 만들고 조회한다.
7. 사용자는 데이터 상태를 바꾼다.
8. 로그아웃하면 보호된 페이지에 들어갈 수 없다.

### 6. 화면 또는 결과물

- 로그인 화면
- 로그아웃 버튼
- 로그인 전후 다르게 보이는 메뉴
- 데이터 목록 화면
- 데이터 상세 화면
- 상태 변경 버튼

### 7. 확인 방법

- 로그인 전에는 보호 페이지가 막히는지 확인한다.
- 로그인 후에는 보호 페이지가 열리는지 확인한다.
- 로그아웃 후 다시 막히는지 확인한다.
- 서로 연결된 데이터가 잘 보이는지 확인한다.

### 8. 조심할 점

인증은 “누구인지 확인”하는 것이다.  
인가는 “무엇을 해도 되는지 확인”하는 것이다.  
둘은 비슷해 보이지만 다르다.

## 문제 4. FastAPI 기반 CRUD 웹 서비스 구축

### 1. 무엇을 만드는가

메모, 할 일, 도서 같은 데이터를 만들고, 보고, 고치고, 지우는 웹 서비스를 만든다.  
이 네 가지 기능을 CRUD라고 한다.

### 2. 왜 필요한가

대부분의 웹 서비스는 데이터를 다룬다.  
쇼핑몰은 상품을 등록하고, 게시판은 글을 쓰고, 일정 앱은 할 일을 저장한다.  
그래서 CRUD는 웹 서비스의 가장 기본 기능이다.

### 3. 쉬운 비유

공책에 메모를 쓰는 것과 같다.

- 쓰기: 새 메모 작성
- 읽기: 메모 보기
- 고치기: 메모 수정
- 지우기: 메모 삭제

### 4. 필요한 부품

- FastAPI 서버
- SQLite 데이터베이스
- SQLAlchemy
- Jinja2 HTML 화면
- 라우터
- 서비스
- 저장소
- 모델

### 5. 동작 순서

1. 사용자가 홈 화면에 들어간다.
2. 목록 화면에서 저장된 데이터를 본다.
3. 등록 화면에서 새 데이터를 입력한다.
4. 서버가 데이터를 DB에 저장한다.
5. 저장 후 목록 화면으로 이동한다.
6. 상세 화면에서 하나의 데이터를 본다.
7. 수정 화면에서 내용을 고친다.
8. 삭제 버튼으로 데이터를 지운다.

### 6. 화면 또는 결과물

- 홈 화면
- 목록 화면
- 상세 화면
- 등록 화면
- 수정 화면
- 삭제 기능

### 7. 확인 방법

- 새 글을 등록하면 목록에 나오는지 확인한다.
- 글을 수정하면 바뀐 내용이 보이는지 확인한다.
- 글을 삭제하면 목록에서 사라지는지 확인한다.
- 새로고침해도 같은 글이 두 번 저장되지 않는지 확인한다.

### 8. 조심할 점

등록 후 바로 같은 화면에 머물면 새로고침할 때 같은 데이터가 또 저장될 수 있다.  
그래서 저장 후 목록 화면으로 이동하는 PRG 방식을 사용한다.

## 문제 5. SQL로 만드는 나만의 데이터베이스

### 1. 무엇을 만드는가

도서 대여, 쇼핑몰, 학급 관리 같은 주제를 정하고, 데이터를 저장할 표를 직접 만든다.  
그리고 SQL로 데이터를 넣고, 찾고, 고치고, 지운다.

### 2. 왜 필요한가

데이터를 엑셀처럼 한 줄로만 쌓으면 중복이 많아지고 실수가 생긴다.  
데이터베이스는 관련 있는 정보를 여러 표로 나누고, 서로 연결해서 정확하게 관리한다.

### 3. 쉬운 비유

학교 도서관을 생각하면 된다.  
학생 명단, 책 목록, 대여 기록을 한 표에 모두 넣으면 복잡하다.  
그래서 학생 표, 책 표, 대여 표로 나누고 서로 번호로 연결한다.

### 4. 필요한 부품

- 테이블 4개 이상
- 기본키
- 외래키
- 제약조건
- 샘플 데이터
- 조회 SQL
- 조인 SQL
- 집계 SQL
- 인덱스

### 5. 동작 순서

1. 주제를 정한다.
2. 필요한 데이터를 표로 나눈다.
3. 각 표의 대표 번호를 정한다.
4. 표끼리 연결할 번호를 정한다.
5. CREATE TABLE로 표를 만든다.
6. INSERT로 샘플 데이터를 넣는다.
7. SELECT로 데이터를 조회한다.
8. JOIN으로 여러 표를 연결해 본다.
9. GROUP BY로 합계나 개수를 구한다.

### 6. 화면 또는 결과물

- 스키마 SQL 파일
- 샘플 데이터 SQL 파일
- 쿼리 SQL 파일
- 실행 결과 캡처
- 선택 사항: ERD 그림

### 7. 확인 방법

- 테이블이 4개 이상인지 확인한다.
- 외래키가 2개 이상인지 확인한다.
- 테이블당 데이터가 10개 이상인지 확인한다.
- JOIN 결과가 사람이 이해할 수 있게 나오는지 확인한다.
- UPDATE와 DELETE가 조건을 가지고 실행되는지 확인한다.

### 8. 조심할 점

DELETE나 UPDATE를 할 때 조건을 빼면 많은 데이터가 한 번에 바뀔 수 있다.  
항상 WHERE 조건을 확인해야 한다.

## 문제 6. React 핵심 개념 마스터: SPA 서비스 구현

### 1. 무엇을 만드는가

React로 여러 화면이 있는 웹 앱을 만든다.  
페이지를 새로고침하지 않고 화면이 바뀌는 SPA 서비스를 만든다.

### 2. 왜 필요한가

요즘 웹 서비스는 빠르게 움직여야 한다.  
사용자가 버튼을 누를 때마다 전체 페이지를 다시 불러오면 느리다.  
React는 필요한 부분만 바꿔서 더 부드러운 화면을 만든다.

### 3. 쉬운 비유

종이책은 한 장씩 넘겨야 한다.  
하지만 전자칠판은 필요한 글자만 바로 바꿀 수 있다.  
SPA는 전자칠판처럼 화면 일부만 바꾸는 방식이다.

### 4. 필요한 부품

- React
- 라우터
- 페이지 컴포넌트
- 재사용 컴포넌트
- 상태
- 폼
- Supabase 또는 Firebase
- 배포 서비스

### 5. 동작 순서

1. 사용자가 웹 앱에 접속한다.
2. React가 첫 화면을 보여 준다.
3. 메뉴를 누르면 주소와 화면이 바뀐다.
4. 목록 화면에서 데이터를 가져온다.
5. 로딩 중이면 로딩 화면을 보여 준다.
6. 데이터가 있으면 목록을 보여 준다.
7. 사용자가 등록, 수정, 삭제를 한다.
8. 바뀐 상태가 화면에 다시 반영된다.

### 6. 화면 또는 결과물

- 홈 화면
- 목록 화면
- 상세 화면
- 등록 화면
- 수정 화면
- Not Found 화면
- 배포된 웹 주소

### 7. 확인 방법

- 메뉴 이동이 잘 되는지 확인한다.
- 목록과 상세 화면이 연결되는지 확인한다.
- 등록, 수정, 삭제가 실제로 되는지 확인한다.
- 로딩, 에러, 빈 상태 화면이 나오는지 확인한다.
- 배포 후에도 같은 기능이 되는지 확인한다.

### 8. 조심할 점

API Key는 코드에 직접 넣지 말고 환경변수에 넣어야 한다.  
배포할 때도 배포 서비스에 환경변수를 따로 등록해야 한다.

## 문제 7. 웹 기초 완성, 나만의 포트폴리오 구축

### 1. 무엇을 만드는가

HTML, CSS, JavaScript만 사용해서 나를 소개하는 포트폴리오 웹사이트를 만든다.  
React나 Bootstrap 같은 외부 도구는 사용하지 않는다.

### 2. 왜 필요한가

웹 개발의 기본은 HTML, CSS, JavaScript이다.  
이 세 가지를 직접 다룰 줄 알아야 다른 도구를 써도 원리를 이해할 수 있다.

### 3. 쉬운 비유

웹사이트를 집이라고 생각하면 된다.

- HTML은 집의 뼈대
- CSS는 벽지와 인테리어
- JavaScript는 전등 스위치와 자동문

### 4. 필요한 부품

- index.html
- css 폴더
- js 폴더
- images 폴더
- Hero 영역
- About 영역
- Skills 영역
- Projects 영역
- Contact 영역
- Footer 영역

### 5. 동작 순서

1. 사용자가 웹사이트에 접속한다.
2. 상단 메뉴로 각 섹션으로 이동한다.
3. 화면 크기에 따라 배치가 바뀐다.
4. 햄버거 메뉴가 모바일에서 열린다.
5. 다크모드 버튼을 누르면 색상이 바뀐다.
6. GitHub API에서 프로젝트 목록을 가져온다.
7. 문의 폼 입력값을 검사한다.
8. GitHub Pages로 배포한다.

### 6. 화면 또는 결과물

- 자기소개 첫 화면
- 기술 목록
- 프로젝트 카드
- 연락 폼
- 다크모드
- 모바일 메뉴
- GitHub Pages 주소

### 7. 확인 방법

- PC와 모바일에서 화면이 깨지지 않는지 확인한다.
- 메뉴를 누르면 부드럽게 이동하는지 확인한다.
- 다크모드가 저장되는지 확인한다.
- GitHub 프로젝트가 불러와지는지 확인한다.
- 이메일 형식이 틀리면 오류가 나오는지 확인한다.

### 8. 조심할 점

`onclick`을 HTML에 직접 쓰지 않는다.  
JavaScript 파일에서 `addEventListener`로 이벤트를 연결한다.

## 문제 8. Mini Git 구축

### 1. 무엇을 만드는가

진짜 Git의 작은 버전을 만든다.  
파일을 실제로 저장하지는 않지만, 커밋, 브랜치, 로그, 검색 같은 핵심 동작을 흉내 낸다.

### 2. 왜 필요한가

Git을 명령어로만 외우면 어렵다.  
직접 작게 만들어 보면 Git이 사실은 “변경 기록을 연결한 지도”라는 것을 이해할 수 있다.

### 3. 쉬운 비유

커밋은 게임 저장 지점이다.  
브랜치는 여러 갈래 길 중 하나를 가리키는 표지판이다.  
로그는 지나온 저장 지점을 순서대로 보는 것이다.

### 4. 필요한 부품

- 커밋 저장소
- 브랜치 목록
- 현재 위치 HEAD
- 커밋 검색 기능
- 로그 출력 기능
- 조상 찾기 기능
- 최단 경로 찾기 기능
- 명령어 입력 화면

### 5. 동작 순서

1. 사용자가 `INIT`으로 저장소를 만든다.
2. `COMMIT`으로 저장 지점을 만든다.
3. `BRANCH`로 새 길을 만든다.
4. `SWITCH`로 다른 길로 이동한다.
5. `LOG`로 기록을 본다.
6. `PATH`로 두 커밋 사이 길을 찾는다.
7. `ANCESTORS`로 조상 커밋을 찾는다.
8. `SEARCH`로 메시지나 작성자를 검색한다.

### 6. 화면 또는 결과물

```text
mini-git> INIT kang
OK
mini-git> COMMIT "first work"
commit a1b2c3
mini-git> LOG
a1b2c3 first work
```

### 7. 확인 방법

- 초기화 전 명령어를 막는지 확인한다.
- 브랜치를 만들고 이동할 수 있는지 확인한다.
- 로그가 올바른 순서로 나오는지 확인한다.
- 없는 커밋을 찾으면 오류가 나오는지 확인한다.
- 검색 결과가 맞는지 확인한다.

### 8. 조심할 점

정렬 기능에서 Python의 `sorted`나 `list.sort`를 쓰면 안 된다.  
직접 정렬 알고리즘을 만들어야 한다.

## 문제 9. Mini Redis 구축

### 1. 무엇을 만드는가

진짜 Redis의 작은 버전을 만든다.  
글자 데이터를 저장하고, 가져오고, 지우고, 시간이 지나면 자동 삭제되게 한다.

### 2. 왜 필요한가

Redis는 아주 빠른 저장소이다.  
빠른 이유를 이해하려면 해시맵, 연결 리스트, 힙 같은 자료구조를 직접 만들어 보는 것이 좋다.

### 3. 쉬운 비유

Mini Redis는 빠른 사물함이다.  
열쇠 이름을 말하면 바로 물건을 꺼내 준다.  
오래 안 쓴 물건은 버리고, 시간이 지난 물건도 자동으로 버린다.

### 4. 필요한 부품

- 직접 만든 해시맵
- 직접 만든 이중 연결 리스트
- 직접 만든 최소 힙
- 문자열 저장 기능
- TTL 만료 기능
- LRU 제거 기능
- 메모리 제한 기능
- 명령어 입력 화면

### 5. 동작 순서

1. 사용자가 `SET name kang`을 입력한다.
2. 프로그램이 name이라는 칸에 kang을 저장한다.
3. 사용자가 `GET name`을 입력한다.
4. 프로그램이 kang을 보여 준다.
5. 사용자가 `EXPIRE name 10`을 입력한다.
6. 10초 뒤 name은 자동으로 사라진다.
7. 메모리가 가득 차면 가장 오래 안 쓴 데이터를 지운다.

### 6. 화면 또는 결과물

```text
mini-redis> SET name kang
OK
mini-redis> GET name
kang
mini-redis> TTL name
(integer) 8
```

### 7. 확인 방법

- SET 후 GET이 되는지 확인한다.
- DEL 후 GET하면 `(nil)`이 나오는지 확인한다.
- EXPIRE 후 시간이 지나면 사라지는지 확인한다.
- maxmemory를 작게 했을 때 오래 안 쓴 값이 지워지는지 확인한다.
- 너무 큰 값을 넣으면 오류가 나오는지 확인한다.

### 8. 조심할 점

Python의 `dict`, `set`, `collections`를 쓰면 안 된다.  
자료구조를 직접 만들어야 한다.

## 문제 10. 실전 Git 협업 워크플로우

### 1. 무엇을 만드는가

여러 사람이 GitHub에서 함께 작업하는 연습을 한다.  
브랜치를 만들고, PR을 보내고, 리뷰를 받고, 충돌을 해결한다.

### 2. 왜 필요한가

혼자 개발할 때와 팀으로 개발할 때는 다르다.  
여러 사람이 같은 파일을 고치면 충돌이 생길 수 있다.  
그래서 정해진 약속과 기록이 필요하다.

### 3. 쉬운 비유

여러 명이 한 권의 공책을 같이 쓴다고 생각하면 된다.  
각자 복사본에 글을 쓰고, 나중에 선생님에게 확인받아 원본 공책에 붙인다.  
같은 줄을 서로 다르게 고치면 의논해서 하나로 정해야 한다.

### 4. 필요한 부품

- GitHub 저장소
- main 브랜치
- feature 브랜치
- Issue
- Pull Request
- 코드 리뷰
- 충돌 해결 문서
- 트러블슈팅 문서

### 5. 동작 순서

1. 팀원이 저장소에 참여한다.
2. main 브랜치는 직접 고치지 못하게 보호한다.
3. 작업할 내용을 Issue로 만든다.
4. 각자 feature 브랜치를 만든다.
5. 작업 후 커밋한다.
6. PR을 만든다.
7. 다른 팀원이 리뷰한다.
8. 리뷰를 반영한다.
9. 승인 후 main에 합친다.
10. 충돌이 생기면 함께 해결하고 문서로 남긴다.

### 6. 화면 또는 결과물

- GitHub 저장소 주소
- SUBMISSION.md
- CONTRIBUTING.md
- conflict-resolution.md
- troubleshooting-log.md
- git log 그래프
- 팀 결과물

### 7. 확인 방법

- 팀원별 PR이 2개 이상인지 확인한다.
- 팀원별 리뷰가 2개 이상인지 확인한다.
- 본인 PR에서 리뷰 반영이 있는지 확인한다.
- 충돌 해결 기록이 있는지 확인한다.
- git log 그래프가 브랜치 작업을 보여 주는지 확인한다.

### 8. 조심할 점

main 브랜치에 바로 push하면 팀 작업 흐름이 깨진다.  
반드시 feature 브랜치에서 작업하고 PR로 합쳐야 한다.
