# evollard 문제 연결 지도

## 최종 그릇: evollard

`evollard`는 위작 의심 미술품을 검토하고, 감정 근거를 정리해 유료 리포트를 발행하는 서비스다.

## 문제별 재료

| 문제 | 재료 | evollard 기능 |
|---:|---|---|
| 1 | AI API + Git diff | 감정 엔진 코드 변경사항을 커밋/PR 초안으로 자동 문서화 |
| 2 | AWS VPC/EC2/SG/IAM | 고객이 접속 가능한 리포트 서버 배포 |
| 3 | FastAPI 인증/인가/관계 | 로그인 고객만 유료 리포트 조회 |
| 4 | FastAPI CRUD | 감정 신청 CRUD |
| 5 | SQL DB | 작가/작품/감정요청/리포트 관계 설계 |
| 6 | React SPA | 운영자 대시보드와 리포트 발행 화면 |
| 7 | HTML/CSS/JS | evollard 소개와 사례 포트폴리오 |
| 8 | Mini Git | 감정 리포트 버전 그래프 학습 |
| 9 | Mini Redis | 분석 결과 캐시, TTL, LRU 학습 |
| 10 | Git 협업 | 팀 기반 리포트 서비스 개발 증빙 |
| 11 | 파일 가계부 | 리포트 매출, 분석 비용, 환불, 클라우드 비용 기록 |
| 12 | Linux 장애 분석 | 분석 Agent 장애 이슈 리포트 |
| 13 | Bash 관제 | Agent/포트/리소스/로그 자동 점검 |

## 유료 리포트 발행 시나리오

1. 고객이 작품 감정을 신청한다.
2. 운영자가 작품 정보와 이미지를 등록한다.
3. 분석 Agent가 색채, 질감, 서명, 작가 스타일 일치도를 계산한다.
4. FastAPI가 감정 상태를 `submitted -> analyzing -> reviewed -> paid_report_issued`로 바꾼다.
5. React SPA에서 운영자가 리포트 초안을 검수한다.
6. 결제 완료 후 고객은 PDF/HTML 유료 리포트를 조회한다.
7. 수익은 파일 기반 장부와 SQL 리포트에 기록된다.
8. 배포/운영/장애 대응 기록이 제출 증거가 된다.

## 채점자가 보는 완성 포인트

- 문제 5의 SQL은 `paid_report` 테이블로 유료 리포트 발행 결과를 남긴다.
- 문제 3의 FastAPI 인증 예시는 고객/운영자 권한 차이를 보여준다.
- 문제 11의 파일 장부는 유료 리포트 매출과 분석 비용을 기록한다.
- 문제 12/13은 리포트 발행 서버가 실제 운영 중 장애를 만났을 때의 대응력을 보여준다.
- 문제 1/10은 혼자 만든 코드가 아니라 팀 협업과 PR 자동화로 관리되는 제품임을 보여준다.
